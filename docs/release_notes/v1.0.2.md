# RuleDSL SDK v1.0.2

An emergency patch with one purpose: **the v1.0.0 and v1.0.1 binaries did not work on your machine.** They worked only on the machine that built them — and every gate we ran happened to be that machine. This release fixes the defect, adds the gates that make its class impossible to ship again, and corrects the v1.0.1 claim that said otherwise.

## The defect, plainly

The engine loaded its grammar definition at runtime from an absolute filesystem path baked in at build time. That path exists only on the build machine (for v1.0.1: the CI runner; for v1.0.0: the maintainer's machine). On any other machine:

- `ruledslc compile` failed with `grammar load failed: Failed to open grammar file: ...`
- `ax_compiler_create()` failed through the C API — nothing built on the SDK could compile a rule.

Evaluation of already-compiled bytecode was never affected; the defect sat in compiler initialization. But the quickstart begins with a compile, so the practical effect was: the bundle did not work out of the box, anywhere.

Why our own verification missed it: the engine test suite, the determinism evidence, and the bundle "clean-room" all ran on machines where the baked path resolved — the source checkout was present. The clean-room copied the bundle to a fresh directory, but on the same filesystem that satisfied the hidden dependency.

## What changed

- **The grammar is now embedded in the binary at build time.** There is no grammar file to find at runtime; the compiled engine carries its grammar. No filesystem lookup remains.
- **Regression guards in the engine test suite** scan the shipped binaries for any baked grammar-path string; the build fails if one reappears.
- **The bundle clean-room is now genuinely clean:** the engine source checkout is deleted from the CI runner before the shipped bundle is exercised, so a binary that secretly depends on a build-machine path now fails the gate instead of passing it.

## Compatibility

v1.0.2 is a patch release. No ABI change; no new error codes; the language stays `0.9` (the implementation tag for the v1.0 spec). Compiled bytecode is byte-identical for the same input. The determinism contract is unaffected — the defect was in *where the compiler found its grammar*, never in *what it produced*.

## Erratum for v1.0.1

The v1.0.1 note claimed "the bundle is now self-sufficient", verified against shipped binaries on both platforms. That verification ran on machines where the baked path resolved, so the claim was wrong. The v1.0.1 note now carries an erratum pointing here.

## Verify this build

- Validate the bundle: `sha256sum -c manifests/HASHES.txt`
- Confirm the compiler: `ruledslc --version` reports `RuleDSL/1.0.2` with `lang=0.9`
- Prove it on your machine: run the quickstart (`docs/quickstart.md`) — its first step is a compile, which is exactly what v1.0.0/v1.0.1 could not do off the build machine.
